# AI 留痕与产物持久化规则

本文档用于约束 AI 辅助开发过程中的留痕、归档、上下文保存和产物交接。目标是让本次 AI 协作的关键信息可以被后续开发者、另一个 AI、另一个工具或另一个项目继续使用。

本文件不重复规定具体代码分层、接口风格、测试方式和技术栈细节；这些内容由项目内的 `AGENTS.md`、README、设计文档或具体技术规范负责。

---

## 1. 核心目标

| 目标 | 说明 |
| --- | --- |
| 可接续 | 下次换 AI、换工具、换人时，可以快速恢复任务背景 |
| 可追溯 | 能知道需求、Prompt、AI 输出、人工修改和最终产物之间的关系 |
| 可复盘 | 出问题时能回看当时的判断、方案、取舍和验证结果 |
| 可迁移 | 目录结构和规则尽量通用，其他项目也能复用 |
| 不重复 | 不把 `AGENTS.md` 已经规定的工程实现规则再写一遍 |

---

## 2. 文档职责边界

| 文件 / 目录 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| `RULES.md` | AI 留痕、产物持久化、上下文接续、归档规则 | 具体代码风格、项目技术细节 |
| `AGENTS.md` | AI 代理在当前仓库中如何读代码、改代码、跑测试 | 通用留痕体系 |
| `docs/` | 需求、设计、计划、汇报、运维等可读文档 | AI 原始输出全文 |
| `artifacts/` | AI 输出、人工修改、diff、快照、原始材料 | 稳定的项目说明文档 |
| `prompts/` | 可复用 Prompt 模板和重要 Prompt | 普通一次性聊天内容 |
| `decisions/` | 重要技术或项目决策 ADR | 日常任务流水账 |

---

## 3. 总体证据链

AI 辅助开发建议形成以下证据链：

```text
上下文 / 需求
  → Prompt
  → AI 原始输出
  → 人工修改或二次整理
  → diff / review / 决策说明
  → 最终代码或交付物
  → 阶段快照 / 总结
```

不是每个小任务都必须保存完整链路；但重要任务、复杂任务、用户明确要求留痕的任务，应尽量保留完整证据链。

## 4. 目录使用规则

| 内容类型 | 保存位置 | 什么时候使用 | 是否必须 |
| --- | --- | --- | --- |
| 项目背景、关键上下文 | `docs/00_context/` | 新 AI 接手前需要快速了解项目时 | 重要项目必须 |
| 需求文档、澄清记录 | `docs/01_requirements/` | 需求已确认，后续实现要以它为准 | 重要功能必须 |
| 设计文档、接口设计、流程设计 | `docs/02_design/` | 涉及架构、数据库、接口、模块边界时 | 重要功能必须 |
| 开发计划 / Plan | `docs/03_plans/` | 复杂任务、重构、多步骤任务、跨会话任务 | 条件必须 |
| 协作流程、工作流说明 | `docs/03_workflow/` | 记录团队或 AI 协作流程 | 可选 |
| 汇报、阶段总结、演示说明 | `docs/04_reports/` | 需要对外展示或阶段交付时 | 可选 |
| 部署、运维、环境说明 | `docs/05_ops/` | 涉及部署、配置、运维操作时 | 条件必须 |
| 待办事项、后续计划 | `docs/06_todo/` | 当前任务未完成或有后续跟进项 | 条件必须 |
| 原始输入材料 | `artifacts/00_raw/` | 用户给的原始资料、样例、参考材料 | 条件必须 |
| AI 原始输出 | `artifacts/01_ai_outputs/` | 需要审计、复盘、对比或交接时 | 重要任务必须 |
| 人工修改版本 | `artifacts/02_human_edits/` | 基于 AI 输出做了人工整理或改写时 | 条件必须 |
| diff 和 review | `artifacts/03_diff_and_reviews/` | 需要比较 AI 输出和最终结果时 | 条件必须 |
| 里程碑快照 | `artifacts/04_release_snapshots/` | 阶段完成、演示、提交前留档 | 可选 |
| Prompt 模板 | `prompts/` | Prompt 以后还会复用 | 可选 |
| 技术决策 ADR | `decisions/` | 决策会影响后续方向或重复讨论 | 条件必须 |

## 5. Plan 保存规则

Plan 的作用是把 AI 的执行计划持久化，避免计划只留在一次对话里。换 AI、换会话、隔几天回来，都应能通过 Plan 继续推进。

### 5.1 什么时候必须保存 Plan

| 场景 | 是否保存 | 位置 |
| --- | --- | --- |
| 用户明确要求“保存计划 / plan 落盘” | 必须 | `docs/03_plans/` |
| 新增完整业务模块 | 必须 | `docs/03_plans/` |
| 数据库结构变更 + 后端接口开发 | 必须 | `docs/03_plans/` |
| 跨多个模块或多层代码的重构 | 必须 | `docs/03_plans/` |
| 任务需要多轮对话或多天完成 | 必须 | `docs/03_plans/` |
| 简单问答、单文件小修、小范围文档润色 | 不强制 | 无 |

### 5.2 Plan 命名

```text
YYYYMMDD_plan_任务名称_v版本号.md
```

示例：

```text
20260702_plan_借阅模块开发_v1.md
20260702_plan_AI留痕规则整理_v1.md
```

### 5.3 Plan 建议结构

| 部分 | 说明 |
| --- | --- |
| 背景 | 为什么要做，来自哪个需求或问题 |
| 当前状态 | 已有什么、缺什么、相关文件在哪里 |
| 目标 | 完成后应该达到什么状态 |
| 范围 | 本次做什么、不做什么 |
| 执行步骤 | 分步骤列出实现路径 |
| 产物清单 | 会生成或修改哪些文档、代码、脚本 |
| 验证方式 | 如何确认完成，运行什么命令或检查什么结果 |
| 风险和注意事项 | 可能踩坑点、依赖、环境限制 |
| 接续提示 | 给下一个 AI 或开发者的继续工作说明 |

## 6. AI 输出保存规则

AI 输出是否保存，取决于它对后续是否有价值。

| 输出类型 | 保存位置 | 保存方式 |
| --- | --- | --- |
| 完整对话导出 | `artifacts/01_ai_outputs/chat_transcripts/` | 原文保存，不改写 |
| AI 生成的方案草稿 | `artifacts/01_ai_outputs/` | 原文保存，可另存整理版 |
| AI 生成的代码草稿 | `artifacts/01_ai_outputs/codegen_drafts/` | 保存关键版本，不保存无价值试错 |
| AI 评审意见 | `artifacts/01_ai_outputs/eval_notes/` | 保存原始意见和时间 |
| 人工整理后的版本 | `artifacts/02_human_edits/` | 标明基于哪个 AI 输出修改 |
| AI 与人工版本差异 | `artifacts/03_diff_and_reviews/` | 保存 diff、评审意见、取舍理由 |

原则：

- `artifacts/01_ai_outputs/` 中的原始输出保存后不要改写。
- 如果需要修改，另存到 `artifacts/02_human_edits/`。
- 不必保存每一次无效尝试，但影响最终结果的关键输出要保存。
- 涉及隐私、账号、Token、日志、截图时，保存前必须脱敏。

## 7. Prompt 保存规则

Prompt 分两类：

| 类型 | 是否保存 | 保存位置 |
| --- | --- | --- |
| 一次性 Prompt | 不强制 | 可不保存 |
| 可复用 Prompt | 建议保存 | `prompts/` |
| 影响关键产物的 Prompt | 建议保存 | `prompts/` 或 `artifacts/01_ai_outputs/` |
| 项目固定工作流 Prompt | 必须保存 | `prompts/` |

Prompt 文件应尽量包含：

- 使用场景
- 输入变量
- 输出格式要求
- 示例
- 适用限制

---

## 8. 决策保存规则

当一次讨论产生了会影响后续工作的选择，应保存为 ADR 或决策记录。

| 场景 | 保存位置 |
| --- | --- |
| 技术栈选择 | `decisions/` |
| 数据库方案选择 | `decisions/` |
| 目录结构调整 | `decisions/` |
| API 契约确定 | `decisions/` |
| AI 工作流规则调整 | `decisions/` |

ADR 建议包含：

- 背景
- 可选方案
- 最终决策
- 选择理由
- 后续影响

## 9. 上下文接续规则

为了让下一个 AI 能继续工作，重要任务结束时应留下接续信息。

建议记录：

| 信息 | 推荐位置 |
| --- | --- |
| 当前任务做到哪一步 | 对应 Plan 或 `docs/06_todo/` |
| 修改了哪些文件 | Plan、报告或最终说明 |
| 哪些命令跑过，结果如何 | Plan、报告或 artifacts |
| 哪些问题没解决 | `docs/06_todo/` 或 Plan |
| 下一个 AI 应该先读什么 | Plan 的“接续提示” |

如果任务很复杂，最终应形成一份简短的接续说明，放在对应 Plan 文件底部。

## 11. 安全与脱敏

- 不保存真实密码、Token、Cookie、私钥、生产数据库连接。
- 截图、日志、AI 对话、测试数据在归档前必须检查敏感信息。
- 如果必须保留敏感字段结构，使用占位符，例如 `<DB_PASSWORD>`、`<TOKEN>`。
- 大文件、二进制文件、录屏等不适合直接进 Git 时，应记录外部存储位置和访问方式。

---

## 12. 最小执行原则

为避免留痕变成负担，按任务重要程度选择留痕级别。

| 任务级别 | 留痕要求 |
| --- | --- |
| 小任务 | 最终说明即可，不强制落盘 |
| 中等任务 | 保存 Plan 或关键设计之一 |
| 复杂任务 | 保存需求、Plan、关键 AI 输出、验证结果 |
| 里程碑任务 | 保存完整证据链和阶段快照 |
| 用户明确要求留痕 | 按用户要求保存 |
