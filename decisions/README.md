# 决策记录目录 (decisions/)

本目录存放项目的架构决策记录（ADR - Architecture Decision Records）。

## 什么是 ADR？

ADR 是一种轻量级的文档格式，用于记录项目中的重要技术决策。每个 ADR 描述一个特定的决策，包括背景、考虑的方案、最终选择及其理由。

## 目录结构

```
decisions/
├── README.md                    # 本文件
├── ADR-0001-scope.md           # 项目范围决策
├── ADR-0002-storage.md         # 存储方案决策
├── ADR-0003-framework.md       # 框架选型决策
└── ...
```

## ADR 命名规范

```
ADR-[序号]-[简短标题].md
```

- 序号：4 位数字，从 0001 开始
- 标题：使用小写字母和连字符，简洁描述决策主题

## ADR 模板

```markdown
# ADR-XXXX: [决策标题]

## 状态
[提议中 | 已接受 | 已废弃 | 已取代]

## 背景
[描述决策的背景和上下文]

## 决策
[描述做出的决策]

## 方案对比

### 方案 A: [方案名称]
- 优点：...
- 缺点：...

### 方案 B: [方案名称]
- 优点：...
- 缺点：...

## 选择理由
[解释为什么选择这个方案]

## 后果
[描述这个决策带来的影响]

## 后续行动
- [ ] 行动项 1
- [ ] 行动项 2
```

## ADR 状态说明

| 状态 | 说明 |
|------|------|
| 提议中 | 决策正在讨论中，尚未最终确定 |
| 已接受 | 决策已被团队接受并实施 |
| 已废弃 | 决策不再适用，但保留历史记录 |
| 已取代 | 决策被新的 ADR 取代，需注明取代者 |

## 为什么要写 ADR？

1. **知识传承**：新成员可以快速了解项目的技术决策
2. **避免重复讨论**：记录决策理由，避免反复讨论同一问题
3. **审计追溯**：提供决策的完整历史记录
4. **培训材料**：可作为最佳实践的教学案例

## 何时需要写 ADR？

- 选择技术栈或框架
- 确定系统架构
- 选择第三方服务
- 定义重要的设计模式
- 任何影响项目方向的重大决策
