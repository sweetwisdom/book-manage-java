# 1. 不要字段注入（必改）

现在：

```java
@Autowired
private UserMapper userMapper;
```

推荐：

```java
@Service
public class UserServiceImpl
        extends ServiceImpl<UserMapper, User>
        implements UserService {

    private final UserMapper userMapper;

    public UserServiceImpl(UserMapper userMapper) {
        this.userMapper = userMapper;
    }
}
```

或者直接：

```java
@RequiredArgsConstructor
@Service
public class UserServiceImpl
        extends ServiceImpl<UserMapper, User>
        implements UserService {

    private final UserMapper userMapper;
}
```

（Lombok）

优点：

* 不会出现空指针注入问题
* 方便单元测试
* Spring 官方推荐

# 2. 既然继承 ServiceImpl，就不要再注入 Mapper（建议改）

你现在：

```java
extends ServiceImpl<UserMapper, User>
```

又：

```java
@Autowired
private UserMapper userMapper;
```

重复了。

因为：

```java
ServiceImpl
```

内部已经有：

```java
protected M baseMapper;
```

所以：

```java
userMapper.selectById(id)
```

可以改：

```java
baseMapper.selectById(id)
```

甚至：

```java
this.getById(id)
```

更符合 MP 风格。

例如：

```java
@Override
public UserVO getUserById(Long id) {

    User user = this.getById(id);

    return user == null ? null : convertToVO(user);
}
```

# 3. createUser 应该加事务（建议改）

现在：

```java
@Override
public UserVO createUser(UserDTO userDTO)
```

如果未来变成：

```java
插入用户
插入角色
插入日志
发送消息
```

中间失败会脏数据。

建议：

```java
@Transactional
@Override
public UserVO createUser(UserDTO userDTO)
```

# 4. DTO -> Entity 转换抽出来（建议改）

现在：

```java
User user = new User();

user.setName(userDTO.getName());
user.setAge(userDTO.getAge());
```

未来字段一多：

```java
20个字段
30个字段
```

会很恶心。

建议：

```java
private User convertToEntity(UserDTO dto) {

    User user = new User();

    user.setName(dto.getName());
    user.setAge(dto.getAge());

    return user;
}
```

然后：

```java
User user = convertToEntity(userDTO);
```

# 5. 不要返回 null（企业里经常被喷）

现在：

```java
if (user == null) {
    return null;
}
```

Controller：

```java
userService.getUserById(id)
```

很容易：

```java
NullPointerException
```

建议：

```java
throw new RuntimeException("用户不存在");
```

或者自定义：

```java
throw new BusinessException("用户不存在");
```

企业项目一般会统一异常处理：

```java
@RestControllerAdvice
```

# 6. convertToVO 应独立成 Converter（推荐）

现在：

```java
private UserVO convertToVO(User user)
```

项目小没问题。

项目大后：

```text
UserDTO
UserVO
UserEntity

OrderDTO
OrderVO
OrderEntity

BookDTO
BookVO
BookEntity
```

会满天飞。

通常抽：

```java
UserConverter
```

例如：

```java
@Component
public class UserConverter {

    public UserVO toVO(User user) {
        ...
    }
}
```

# 我会改成这样

核心结构：

```java
@Service
@Transactional
@RequiredArgsConstructor
public class UserServiceImpl
        extends ServiceImpl<UserMapper, User>
        implements UserService {

    @Override
    public UserVO getUserById(Long id) {

        User user = this.getById(id);

        if (user == null) {
            throw new BusinessException("用户不存在");
        }

        return convertToVO(user);
    }

    @Override
    public UserVO createUser(UserDTO dto) {

        User user = convertToEntity(dto);

        this.save(user);

        return convertToVO(user);
    }
}
```

对于你这个 Demo 项目，我给分的话：

```text
当前版本：7/10
学习阶段：很好
企业项目：需要重构 Mapper使用方式、异常处理、事务管理
```

最值得先改的 3 个地方：

1. 去掉 `@Autowired UserMapper`
2. 使用 `this.save()/this.getById()/this.list()`
3. 引入统一异常处理，不返回 `null`

这三个改完，就已经比较接近 RuoYi 这类项目的 Service 写法了。
