这个问题其实是 **Java 集合框架设计导致的**。

作为 JS 开发者，你可以先这样理解：

### JS 里面

```js
const a = { name: "Tom" }
const b = { name: "Tom" }

console.log(a === b) // false
```

JS 对象比较的是内存地址。

而 Java 默认也是这样：

```java
User a = new User("Tom");
User b = new User("Tom");

System.out.println(a.equals(b)); // false
```

因为默认 `equals()` 最终也是比较地址。

---

## 为什么要重写 equals

假设你觉得：

```java
name 相同就是同一个用户
```

那么你会重写：

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User)) return false;

    User user = (User) o;
    return Objects.equals(name, user.name);
}
```

此时：

```java
User a = new User("Tom");
User b = new User("Tom");

a.equals(b); // true
```

看起来没问题。

---

## 但是 HashSet 出问题了

来看：

```java
Set<User> set = new HashSet<>();

set.add(new User("Tom"));
set.add(new User("Tom"));

System.out.println(set.size());
```

你期望：

```java
1
```

因为两个对象 equals 返回 true。

结果：

```java
2
```

为什么？

---

## HashSet 根本不先调用 equals

很多新手误解这一点。

HashSet 流程：

```text
1. 先算 hashCode
2. 找桶(bucket)
3. 同一个桶里再调用 equals
```

类似：

```text
hashCode -> 楼层
equals   -> 房间号
```

先找到楼层，再挨个比较房间。

---

假设：

```java
a.equals(b) == true
```

但是：

```java
a.hashCode() == 100
b.hashCode() == 200
```

那么：

```text
100号桶
200号桶
```

它们根本不会碰面。

HashSet 会认为：

```text
这是两个不同对象
```

所以 size=2。

---

## Java 规定的契约（Contract）

Java 官方要求：

```text
如果两个对象 equals 返回 true

那么它们的 hashCode 必须相同
```

即：

```java
a.equals(b) == true
```

必须保证：

```java
a.hashCode() == b.hashCode()
```

否则 HashMap、HashSet 全部会出现异常行为。

---

## 所以必须同时重写

错误写法：

```java
@Override
public boolean equals(Object o) {
    ...
}
```

没有重写 hashCode。

正确写法：

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User)) return false;

    User user = (User) o;
    return Objects.equals(name, user.name);
}

@Override
public int hashCode() {
    return Objects.hash(name);
}
```

---

## 用 JS Map 类比

JS 的 Map：

```js
const map = new Map()

map.set({id:1}, "A")

console.log(map.get({id:1}))
// undefined
```

因为 JS Map 也是按引用地址比较。

Java 的 HashMap 更像：

```js
const key = JSON.stringify(obj)
```

先计算一个哈希值：

```js
hash(obj)
```

然后再决定是否相等。

所以 Java 必须保证：

```text
equals 认为一样
↓
hashCode 也必须一样
```

否则 HashMap 无法工作。

---

## 实际开发建议

IDEA 里直接生成：

```java
Generate
    -> equals() and hashCode()
```

或者用：

```java
@Data
```

（[Project Lombok](https://projectlombok.org/?utm_source=chatgpt.com)）

自动生成：

```java
@EqualsAndHashCode
```

企业项目里几乎不会手写复杂的 `equals/hashCode`，通常交给 Lombok 或 IDEA 生成。

记一句最重要的规则：

> 重写 equals 后，凡是参与 equals 比较的字段，也必须参与 hashCode 计算。否则 HashMap、HashSet 会出现逻辑错误。
---

如果是我自己写业务代码，我一般会遵循一个原则：

```text
业务主键参与 equals
业务主键参与 hashCode
```

比如用户表：

```sql
user
├── id
├── username
├── email
```

通常 `id` 是主键，那么：

```java
import java.util.Objects;

public class User {

    private Long id;
    private String username;
    private String email;

    public User(Long id, String username, String email) {
        this.id = id;
        this.username = username;
        this.email = email;
    }

    @Override
    public boolean equals(Object o) {
        // 同一个对象
        if (this == o) {
            return true;
        }

        // null 或类型不同
        if (o == null || getClass() != o.getClass()) {
            return false;
        }

        User user = (User) o;

        // 业务主键比较
        return Objects.equals(id, user.id);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id);
    }
}
```

测试：

```java
User u1 = new User(1L, "Tom", "a@qq.com");
User u2 = new User(1L, "Jerry", "b@qq.com");

System.out.println(u1.equals(u2));
```

输出：

```text
true
```

因为业务上：

```text
id = 1
就是同一个用户
```

---

## 为什么不用 username + email

很多新人喜欢：

```java
@Override
public boolean equals(Object o) {
    return Objects.equals(username, other.username)
        && Objects.equals(email, other.email);
}
```

然后：

```java
@Override
public int hashCode() {
    return Objects.hash(username, email);
}
```

理论上没问题。

但是：

```java
user.setUsername("newName");
```

字段变了。

而对象已经放进：

```java
HashSet
HashMap
```

此时 hashCode 也变了。

会出现诡异问题：

```java
set.contains(user);
```

返回：

```text
false
```

虽然对象明明就在集合里。

因为桶位置已经变了。

---

企业项目里经常踩这个坑：

```java
@Data
public class User {
    private Long id;
    private String username;
    private String email;
}
```

Lombok 默认：

```java
@EqualsAndHashCode
```

会把所有字段都参与计算。

结果：

```java
user.setUsername(...)
```

之后 HashSet 找不到自己。

所以很多项目会改成：

```java
@Getter
@Setter
@EqualsAndHashCode(of = "id")
public class User {
    private Long id;
    private String username;
    private String email;
}
```

或者：

```java
@Data
@EqualsAndHashCode(of = "id")
public class User {
    private Long id;
    private String username;
    private String email;
}
```

---

再结合你 JS 背景理解：

JS：

```js
const set = new Set()

const user = {
  id: 1,
  name: "Tom"
}

set.add(user)

user.name = "Jerry"

set.has(user) // true
```

因为 JS Set 存的是引用地址。

Java：

```java
HashSet<User>
```

存的是：

```text
hashCode -> bucket -> equals
```

如果 hashCode 依赖的字段被修改：

```java
username
email
```

就相当于：

```text
门牌号变了
```

HashSet 找不到原来的位置。

所以 Java 老鸟一般有个经验：

```text
equals/hashCode 只使用不会变化的唯一标识
```

对于数据库实体：

```text
id
UUID
业务编码(code)
```

最安全。
